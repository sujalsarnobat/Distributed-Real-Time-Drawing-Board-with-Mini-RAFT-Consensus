4. Mini-RAFT Specification
A simplified RAFT protocol will be implemented as follows:

4.1 Node States
‚óè‚Äã Follower ‚Äî waits for leader heartbeats‚Äã
‚óè‚Äã Candidate ‚Äî initiates elections‚Äã
‚óè‚Äã Leader ‚Äî handles replication and commits‚Äã

4.2 Election Rules
‚óè‚Äã Election timeout: random 500‚Äì800 ms‚Äã
‚óè‚Äã If follower misses heartbeat ‚Üí becomes candidate‚Äã
‚óè‚Äã Candidate increments term and requests votes‚Äã
‚óè‚Äã Node becomes leader on receiving majority (‚â•2) votes‚Äã

Heartbeat Interval: 150 ms

4.3 Log Replication Rules
When clients send a stroke:
Client ‚Üí Gateway ‚Üí Leader
Leader actions:
1.‚Äã Append stroke to local log‚Äã
2.‚Äã Send AppendEntries(term, leaderId, entry) to followers‚Äã
3.‚Äã Followers append to their logs and respond‚Äã

4.‚Äã When majority acknowledges, leader marks entry as committed‚Äã
5.‚Äã Leader broadcasts stroke to Gateway ‚Üí all clients‚Äã

Safety Rules
‚óè‚Äã Committed entries must never be overwritten‚Äã
‚óè‚Äã Higher term always wins‚Äã
‚óè‚Äã Split votes must retry election‚Äã
‚óè‚Äã A restarted node must catch up via leader ‚Üí follower sync
Catch-Up Protocol (Restarted Nodes)
1.‚Äã Restarted node starts in Follower state with an empty log
2.‚Äã On first AppendEntries from the leader, prevLogIndex check fails ‚Üí follower responds
with its current log length
3.‚Äã Leader calls /sync-log on the follower, sending all committed entries from that index
onward
4.‚Äã Follower appends all missing entries and updates its commit index
5.‚Äã Follower is now in sync and participates normally
